# コミットの方針

指摘が無くコミットに進む際に、これからの変更とブランチの未マージのコミットをどう整えるかを判断するための方針。

- メインブランチに取り込まれていない変更のコミットを、適切な粒度に整える。
- PR には意味のある単位のコミットを含める。
- typo 修正・リファクタリング・テストコード追加などのマイナーな修正は、既存のコミットに amend する。
- 追加方針や方針変更により作業ブランチの内容が大きく変わった場合は、squash して最終的な提出コミットにまとめる。方針変更の過程はコミットに残さない。
- 最終的な提出物が意味的に独立しており、コミットを分割するのが適切な場合にのみ、コミットを追加する。
- **プッシュ済みかどうかは、整理するかどうかの判断材料にしない。** メインブランチに取り込まれるまでは作業中の履歴とみなし、既にプッシュ済みのコミットであっても上記の方針で amend・squash・rebase して整える。プッシュ済みのコミットを整えた場合は `git push --force-with-lease` で更新する。
- 公開 PR 上のレビュー指摘への対応も同様に、原則として対象のコミットに amend/squash して取り込む。「レビュー対応の履歴を残す」ことを理由に別コミットへ分けない（最終的な提出物として意味的に独立している場合を除く）。
- コミットメッセージおよび PR のタイトル・本文の言語は、CLAUDE.md に指定があればそれに従う。指定がなければ英語で書く。
- テストケースを追加した場合は、追加したテストケースの一覧を PR 本文に記載する。目的は追加したテストケースの俯瞰なので、`context` / `example` やテスト関数などテストケース単位で列挙し、各ケースが何を検証するのかが本文だけで分かるようにする。既存テストケースの内容を修正しただけの場合は、この一覧は不要。
- 上記の一覧を載せた PR を更新する際は、追加したテストケースの増減に合わせて一覧を追従させる（テストケースを新たに追加したら追記し、追加していたケースを削除・改名したら該当箇所を修正する）。本文の一覧と実際に追加したテストケースが食い違ったまま残らないようにする。
