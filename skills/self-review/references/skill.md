# スキル（SKILL.md）レビュー観点

`skills/**/*.md`（各スキルの `SKILL.md` と、同梱する任意の Markdown）を変更した場合に、共通観点に加算する。スキルディレクトリ配下の構成は自由で、決まったディレクトリ名はない。観点の出典は `skill-creator` スキルの Skill Writing Guide。

## description（発火の主機構）

Claude は description だけを見てスキルを発火させるか判断する。

- description だけで **「何をするか」と「いつ使うか」の両方**が読み取れるか。
- **トリガー条件（いつ呼ぶか）・呼び出しの動機は description に集約**しているか。本文側に重複して書いていないか。

## 本文の分量と構造（Progressive Disclosure）

- SKILL.md 本文が肥大化していないか（500 行程度を目安）。手順の詳細やドメイン別の内容は別ファイルに分離し、本文からは「いつそれを読むか」を明示して誘導しているか。
- 300 行を超える参照ファイルには目次があるか。

## 記述スタイル

- 指示は**命令形**で書けているか。
- **簡潔か**。役割を果たしていない記述は削る。特に、そのスキルが**いつ・なぜ呼ばれるのかという背景説明を本文に書き込みすぎていないか**（トリガー条件・呼び出しの動機は description の役目）。
- ただし簡潔さを理由に、**各指示の根拠（why）まで削っていないか**。ここでの「削ってよい背景」は「このスキルをなぜ呼ぶか」であって、「この手順がなぜ重要か」は残す。
- 根拠のない `MUST` / `ALWAYS` / `NEVER` の乱用がないか。強い禁止・強制に頼るより、**なぜそれが重要かを説明**してモデルに理解させているか。

## frontmatter

- `name` と `description` があるか（必須）。
- このリポジトリの慣習に沿っているか。
  - `license: Apache-2.0` を付けているか。
  - description は日本語で書いているか（既存スキルに合わせる）。
  - サブエージェントで実行するスキルなら `context` / `agent` を、モデルからの自動起動を禁じるスキルなら `disable-model-invocation: true` を指定しているか。

## 安全性（Principle of Lack of Surprise）

- スキルの内容が、description から想像される意図を裏切っていないか。マルウェア・不正アクセス・情報窃取などを助長する内容が紛れていないか。
