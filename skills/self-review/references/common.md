# 共通レビュー観点

言語を問わず適用する観点。

## Correctness（正しさ）

- 要件を満たしているか。ロジックの誤りは無いか。
- エッジケース（空・境界値・異常系）を考慮しているか。

## Code Quality（コード品質）

- 命名は意図を表しているか。可読性は十分か。
- 不要な重複や過度な複雑さは無いか。

## Comments（コメント）

- What（実装の写経・言い換え）ではなく Why（コードから読み取れない背景・意図・理由）を書いているか。命名で語れるならコメントより命名を改善する。
- コメントとコードが一致しているか。修正したファイルの既存コメントを確認し、今回の改修で陳腐化・矛盾（＝嘘に）していないか。死んだコードのコメントアウトは残さない。
- 改修対象の既存ソースコードを確認し、周囲と同程度のボリューム感（粒度・量）で書いているか。過不足なく、その箇所の慣習に合わせる。

## Testing（テスト）

- 新規・変更した振る舞いにテストがあるか。
- 失敗ケース・境界条件のテストがあるか。
- 価値の薄いテストを書いていないか。コード中に宣言的に書かれている内容（定数の値など）をそのまま検証するテストは、実装の写経に過ぎず価値が低い。振る舞い・ロジックを検証しているか。

## Security（セキュリティ）

- 入力値の検証は適切か。
- 秘密情報の混入や、脆弱性につながる処理は無いか。
