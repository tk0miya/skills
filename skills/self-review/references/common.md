# 共通レビュー観点

言語を問わず適用する観点。

## Correctness（正しさ）

- 要件を満たしているか。ロジックの誤りは無いか。
- エッジケース（空・境界値・異常系）を考慮しているか。

## Code Quality（コード品質）

- 命名は意図を表しているか。可読性は十分か。
- 不要な重複や過度な複雑さは無いか。

## Comments（コメント）

- **最優先：端的な表現になっているか。** 冗長な説明を避け簡潔に書く。改修対象の既存ソースコードを確認し、周囲と同程度のボリューム感（粒度・量）で、その箇所の慣習に合わせているか。
- クラスや関数など、コメント先のコード自身の責務が記述されているか。そのコンポーネントの利用方法・責務の説明を主とし、外部のコンポーネントの責務への言及を避けているか。
- コードを読めば自明な内容を書いていないか。メソッド名やメソッド本体の内容を自然言語に言い換えただけのコメントは不要。命名で語れるならコメントより命名を改善する。
- コメントの追加・改修時に、修正したファイルの既存コメントと整合しているか。重複や無価値な記述を増やさず、今回の改修で陳腐化・矛盾（＝嘘に）した記述を残していないか。死んだコードのコメントアウトも残さない。
- レビューや議論で生じた経緯をコメントに残していないか。経緯はコミットメッセージで表現すれば十分。

## Testing（テスト）

- 新規・変更した振る舞いにテストがあるか。
- 失敗ケース・境界条件のテストがあるか。
- 価値の薄いテストを書いていないか。コード中に宣言的に書かれている内容（定数の値など）をそのまま検証するテストは、実装の写経に過ぎず価値が低い。振る舞い・ロジックを検証しているか。

## Security（セキュリティ）

- 入力値の検証は適切か。
- 秘密情報の混入や、脆弱性につながる処理は無いか。
